PAPER 2: Making Middleboxes Someone Else’s Problem: Network Processing as a Cloud Service
1

Paper summary

The main problem being tackled in this paper is enterprise networks rely on middleboxes (e.g.,
firewalls, proxies) for security and performance, but deploying them in-house is expensive, complex to manage, and prone to failures. Outsourcing to the cloud is challenging because middleboxes typically require an ”on-path” placement to observe bidirectional traffic. Moving them
off-path risks breaking functional equivalence, increasing network complexity, and adding latency.
The key idea is APLOMB (Appliance for Outsourcing Middleboxes), “a practical service for
outsourcing enterprise middlebox processing to the cloud”. This approach leverages cloud computing to reduce infrastructure costs, simplify management, and provide elastic fault-tolerance.
A low-complexity APLOMB gateway at the enterprise boundary tunnels traffic to and from
cloud datacenters. It uses DNS-based redirection to ensure bidirectional traffic traverses the
same latency-optimized cloud point-of-presence (PoP). For location-dependent middleboxes like
WAN optimizers, an augmented version called APLOMB+ embeds traffic compression directly
into the enterprise gateway. A centralized cloud controller manages network policies and dynamically scales middlebox instances based on load.
The system was guided by a “survey of 57 enterprise networks” and evaluated using an
Amazon EC2 implementation with real traffic traces. Results show APLOMB+ can “outsource
over 90% of middlebox hardware”of a large enterprise. In a real-world case study, this cloud
detour imposed an “average latency penalty of 1.1ms and median bandwidth inflation of 3.8%”.
However, the evaluation is limited to a small number of scenarios and may not fully reflect
diverse real-world enterprise environments.
2

Contributions

This paper makes three primary contributions. First, it presents “the first large-scale academic
study of middlebox deployment” across 57 enterprise networks, quantifying their cost, complexity, and failure modes. Second, it introduces APLOMB, a system architecture for outsourcing
enterprise network processing to the cloud using DNS-based redirection and a lightweight gateway, enabling over 90% of middlebox hardware to be outsourced. Third, it demonstrates through
a real-world case study that this approach significantly reduces on-premise infrastructure while
maintaining low latency (1.1ms) and modest bandwidth overhead (3.8%).
3

Problems

While APLOMB presents a compelling architecture, its real-world adoption faces major hurdles
regarding data privacy. The design inherently gives the cloud provider “unencrypted access to
traffic flows”. For enterprises bound by strict security policies or compliance regulations, this
level of exposure is unacceptable. Furthermore, the system doesn’t actually manage to outsource everything. It completely misses ”internal” firewalls, which are crucial for blocking
threats that originate from inside the enterprise itself. The authors also concede that replacing
dedicated location-dependent hardware with their generic APLOMB+ compression module is
a compromise, as it can’t perfectly replicate the behavior of all specialized appliances. On the
financial side, the bandwidth cost model may significantly limit practicality. Because APLOMB
tunnels all traffic to the cloud, it necessitates “paying for bandwidth twice”: once for the enterprise’s own access link and again to the cloud provider. Given that standard cloud providers
“price bandwidth purely by volume”, running this setup without negotiating heavily discounted
wholesale rates could be prohibitively expensive. Additionally, the system introduces dependency on cloud providers, raising concerns about vendor lock-in and performance variability
across regions.
