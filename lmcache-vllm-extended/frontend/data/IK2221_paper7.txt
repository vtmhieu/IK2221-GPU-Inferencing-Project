PAPER 7: Efficient Memory Management for Large Language Model Serving
with PagedAttention
1

Paper summary

A major problem when serving Large Language Models (LLMs) is managing the Key-Value (KV)
cache. This cache grows dynamically, making its final size unpredictable. Existing systems preallocate large, continuous memory blocks, which creates severe fragmentation and limits batch
sizes.
To solve this, the authors propose ”PagedAttention”, “an attention algorithm inspired by the
classical virtual memory and paging techniques in operating systems.” It splits the KV cache
into fixed-size blocks that act like virtual memory pages, removing the need for contiguous
physical memory. The vLLM engine dynamically maps these logical blocks to physical ones.
This approach largely eliminates external fragmentation and minimizes internal fragmentation
to the unused space in the final block.
When tested on ShareGPT and Alpaca datasets, “vLLM improves the throughput of popular
LLMs by 2-4x with the same level of latency compared to the state-of-the-art systems, such as
Faster Transformer and Orca.”
2

Contributions

First, the paper introduces PagedAttention, which makes it possible to achieve “near-zero waste
in KV cache memory”. By dynamically allocating memory blocks instead of reserving contiguous chunks, it fundamentally solves the fragmentation issues that limit LLM batch size.
Second, it presents a flexible KV cache sharing technique for different requests using a “copyon-write mechanism.” This dramatically improves the performance of complex decoding tasks
like beam search and parallel sampling, yielding memory savings of up to 55%.
Finally, the authors built and open-sourced the vLLM engine. This saves a large amount of
computing power and money for hosted LLM services by increasing throughput by 2 to 4 times
over systems like Orca, while maintaining latency.
3

Problems

Despite its benefits, vLLM faces challenges when scaling under resource pressure. When GPU
memory is full, the system swaps evicted blocks to the CPU, but “swapping incurs excessive
overhead with small block sizes.” This happens because it causes “numerous small data transfers
between CPU and GPU, which limits the effective PCIe bandwidth.”
Additionally, the system relies on a single central scheduler to manage block tables and
coordinate distributed GPU workers. This centralized design is a potential single point of
failure and could become a computational bottleneck as cluster sizes increase.
Lastly, setting a static block size creates a strict trade-off: large blocks cause internal fragmentation for short sequences, while small blocks underutilize GPU parallelism. Furthermore,
applying these techniques to standard DNN training or compute-bound inference “may rather
degrade the performance due to the extra overhead of memory indirection and non-contiguous
block memory.”
