PAPER 3: The Click Modular Router
1

Paper summary

Modern routers must perform complex tasks like prioritizing traffic, network address translation,
and firewalling. However, traditional router software has “closed, static, and inflexible designs”,
making it extremely difficult to add new features or modify existing behaviors.
The authors propose the Click Modular Router, a flexible software architecture that builds
routers by composing small, distinct “packet processing modules called elements”.
A Click router’s configuration is built as a “directed graph with elements at the vertices”,
where packet flows act as edges. Packet movement is controlled by two connection types:
“push” processing, which handles incoming packets from hardware, and “pull” processing, which
controls transmission timing and scheduling. Furthermore, Click uses explicit queue elements
rather than hiding packet storage, giving developers precise control over routing queues.
The authors tested Click by implementing a compliant IP router on standard PC hardware
running a modified Linux kernel. To handle heavy traffic without crashing, they replaced
standard hardware interrupts with a polling mechanism. The evaluation showed the Click router
could achieve a “maximum loss-free forwarding rate of 333,000 64-byte packets per second”
—about four times faster than standard Linux routing—proving that a highly modular design
is fully compatible with excellent performance.
2

Contributions

The paper proposes a highly modular software architecture for building routers from finegrained, composable packet-processing components called elements. This makes it possible
to easily configure and extend a router’s forwarding path, allowing developers to add complex features like dropping policies or Differentiated Services simply by adding or rearranging
elements in a configuration graph.
Additionally, the authors present a new technique for managing control flow using distinct
push and pull connections, combined with “flow-based router context”. This makes it possible to
cleanly model both unsolicited hardware-driven packet arrivals and carefully timed transmission
schedules, enabling the use of explicit routing queues and composable packet schedulers.
Finally, the paper introduces a pure polling-based device handling mechanism that replaces
standard hardware interrupts. This dramatically improves the performance of software routers
on standard PC hardware by eliminating “receive livelock” , allowing the Click IP router to
achieve a “maximum loss-free forwarding rate of 333,000 64-byte packets per second”.
3

Problems

While the Click Modular Router offers significant flexibility, it faces several scaling limitations.
Because the architecture organizes around packet flow rather than connections, Click cannot
schedule the CPU per individual flow.
Operating on a single thread means “any packet transfer method (push or pull) must return
to its caller before another task can begin”. Moreover, modularity introduces overhead, as
inter-element packet transfers cause “either one or two virtual function calls”, consuming extra
CPU cycles.
Finding elements via “flow-based router context” can also lack specificity, mistakenly including error-handling elements. Additionally, the configuration language lacks variables, forcing
data duplication across files, and developers cannot “dynamically add a new kind of annotation”
to the packet data structure.
Finally, achieving peak performance relies on a “pure polling system”, necessitating custom
modifications to the Linux kernel’s network drivers—an unrealistic deployment requirement for
many standard environments.
