PAPER 9: Janus: A Unified Distributed Training Framework for Sparse
Mixture-of-Experts Models
1

Paper summary

Training large Mixture-of-Expert (MoE) models is very costly. This is mainly because of the
huge “All-to-All” communication bottleneck in the traditional “expert-centric” method, where
the system keeps “experts in-place and exchanging intermediate data to feed experts”. To solve
this, the authors present a new “data-centric” approach in their system called Janus. Instead
of moving data, Janus works by “keeping data in-place and moving experts between GPUs”.
Since “experts’ size can be smaller than the size of data”, this greatly reduces network traffic.

By using this pull-based model, Janus can fetch experts one by one. This allows for “finegrained asynchronous communication”, meaning the system can calculate data and fetch experts
at the same time to hide network delays. Janus also improves performance by caching remote
experts in the CPU memory so local GPUs can share them. Furthermore, it uses a “topologyaware priority strategy” to stop local link traffic jams and allows “experts to be prefetched”
when the network is not busy. When tested on a 32-GPU A100 cluster compared to Microsoft’s
Tutel, Janus could “reduce the traffic up to 16x and achieves up to 2.06 speedup”.
2

Contributions

First, the paper introduces a new “data-centric” communication method for MoE models. By
pulling experts to the data rather than sending tokens to experts, it dramatically cuts down
network traffic and balances the workload. This successfully avoids the severe “All-to-All communication” bottleneck in large MoE training.
Second, the authors design a “fine-grained” asynchronous scheduling system with “hierarchical communication”. Breaking requests into smaller tasks lets a GPU compute one expert
while downloading another. Also, saving remote experts in the CPU memory for local sharing
heavily reduces expensive cross-node traffic.
Third, the work provides a topology-aware priority scheduler and a prefetching strategy.
The system cleverly manages expert downloads across different hardware links to avoid traffic
jams. It also uses idle network time to pre-download experts, which boosts the overall training
speed.
3

Problems

First, the system’s scalability drops as the cluster grows. The efficiency gain formula is R =
BSk
4nHE . Because the number of machines (n) is at the bottom of the fraction, the speed benefit
naturally decreases when adding more machines. This means the method might struggle in
extremely large superclusters unless combined with other techniques like “tensor parallelism”.
Second, the approach makes strong assumptions about how tokens are distributed. The
mathematical proof assumes tokens are evenly balanced among experts to compare it with the
expert-centric baseline. In rare cases where token routing is highly unbalanced (for example,
if one worker sends almost all tokens to its own local experts), the old expert-centric method
might actually create less network traffic.
Third, the system could face issues if expert sizes grow too fast. While the data-centric
method stops memory crashes caused by routing huge token batches, it creates a new problem: it
must constantly download and cache whole expert modules on local GPUs. If future models have
much larger expert sizes, the limited GPU memory will become a major bottleneck, requiring
tensor parallelism to split the matrices.
