LLM Inference Beyond a Single Node: From Bottlenecks to Mitigations with Fast All-Reduce Communication
Authors: Prajwal Singhania, Siddharth Singh, Lannie Dalton Hough, Akarsh Srivastava, Harshitha Menon, Charles Fredrick Jekel, Abhinav Bhatele
 
**Abstract**
As LLMs grow too large for a single GPU, multi-node distributed inference becomes necessary, but introduces new bottlenecks that single-node optimisations do not address. This paper presents a systematic performance study of multi-node LLM inference across two model-parallel strategies — tensor parallelism (TP) and hybrid tensor-pipeline parallelism (HP) — using state-of-the-art engines (vLLM, SGLang) alongside YALIS, a new research-oriented prototype engine. The study identifies all-reduce communication as a critical bottleneck for TP in decode-heavy workloads, where NCCL performs poorly at the small message sizes that characterise this regime. To address this, the authors introduce NVRAR, a hierarchical all-reduce algorithm built on NVSHMEM using recursive doubling, which achieves 1.9x–3.6x lower latency than NCCL for 128 KB–2 MB messages and up to 1.72x lower end-to-end batch latency for the Llama 3.1 405B model in multi-node decode-heavy workloads.
 
**1. Introduction**
LLMs are growing in size, context length, and compute demand due to trends including longer sequences, reasoning architectures, and test-time scaling. Models such as Llama 3.1 405B exceed the memory of a single node's GPUs, requiring multi-node execution. Single-node inference has been extensively studied and optimised, but multi-node inference is comparatively underexplored. The key challenge is that inter-node interconnects are much slower than intra-node NVLink, so parallelisation strategies that work well within a node can degrade substantially when crossing node boundaries. This paper addresses three questions: how do TP and HP scale across multiple nodes; what are the bottlenecks; and can collective communication be optimised to address them?
 
**2. Background**
LLM inference consists of a compute-bound prefill phase (processing all input tokens in one forward pass) and a memory-bandwidth-bound decode phase (generating output tokens one at a time with frequent KV-cache accesses). In pipeline parallelism (PP), contiguous layer groups are assigned to pipeline stages with point-to-point communication; in tensor parallelism (TP), each layer's computation is split across GPUs, requiring an all-reduce per layer to aggregate partial results. NCCL, the standard GPU communication library, implements two all-reduce strategies: Ring (linear cost in number of nodes, good for large messages) and Tree (logarithmic cost, lower latency for small messages). NVSHMEM is NVIDIA's OpenSHMEM-based library supporting GPU-initiated one-sided communication, which underpins NVRAR.
 
**3. Performance Study of Multi-Node Inference**
The authors introduce YALIS (Yet Another LLM Inference System), a prototype inference engine built for instrumented multi-node experimentation on HPC Slurm environments. It uses LitGPT for model definitions, Torch Compile and CUDA Graphs for low-overhead execution, and AxoNN for tensor parallelism. Experiments run Llama 3.1 70B and 405B on the Perlmutter supercomputer (A100 GPUs, HPE Slingshot-11 interconnect) and the Vista system (GH200 GPUs, InfiniBand), with two workload types: prefill-heavy (long prompts, short outputs) and decode-heavy (shorter prompts, long outputs).
 
The scaling study reveals two key findings. First, neither TP nor HP scales well — batch latency does not decrease inversely with GPU count. For TP, latency improvements plateau or reverse beyond two nodes for both models and both workloads. For HP, decode-heavy workloads scale poorly because pipeline parallelism micro-batches do not reduce matrix multiplication time in the decode regime: splitting the batch dimension (as PP does) gives no speedup for small-M GEMMs typical of decode, whereas splitting the weight dimension (as TP does) does help. Second, the relative merits of the two strategies are workload-dependent: HP outperforms TP for compute-bound prefill-heavy workloads due to lower communication overhead, while TP outperforms HP for memory-bound decode-heavy workloads despite its communication cost.
 
Performance breakdowns show that in TP, communication time grows by ~1.6x when scaling from 8 to 16 GPUs for decode-heavy workloads, offsetting the gains from reduced computation. Benchmarking NCCL against GPU-aware MPI on Perlmutter confirms the root cause: for small messages (128 KB–1 MB, characteristic of decode-phase all-reduces), NCCL is 1.5–2x slower than Cray-MPICH and its latency scales more steeply with node count. MPI's advantage in this regime is attributed to its use of recursive-doubling algorithms, which scale logarithmically rather than linearly.
 
**4. NVRAR: Hierarchical Recursive All-Reduce**
NVRAR is a three-phase hierarchical all-reduce designed for the small-message, latency-bound regime of decode-phase TP. The three phases are: (1) intra-node reduce-scatter using NCCL's reduce-scatter (fast NVLink), which reduces each GPU's share from |M| to |M|/G bytes; (2) inter-node recursive-doubling all-reduce using NVSHMEM non-blocking put primitives, completing in log2(N) steps where each GPU communicates only with its XOR-paired peer at each step; and (3) intra-node all-gather using NCCL's all-gather to reconstruct the full tensor on every GPU.
 
Three optimisations reduce latency in the inter-node phase. First, chunked non-blocking communication divides each message into small chunks processed by independent thread blocks, overlapping computation and communication across GPU SMs and enabling asynchronous progress. Second, fused data-flag payloads combine 4 bytes of data with a 4-byte synchronisation flag into a single 8-byte payload, enabling atomic ordered delivery and per-warp reduction immediately on receipt without explicit signal primitives — which introduce non-trivial overhead on Slingshot due to software rather than hardware fences. Third, sequence-number-based global synchronisation assigns each all-reduce a unique sequence number; GPUs wait only for their direct peers to reach the same sequence number before proceeding, avoiding a global barrier and allowing a GPU to reuse its result immediately while its peers finish.
 
Theoretically, NVRAR scales as O(log N) in the number of nodes, matching Tree all-reduce asymptotically but with a lower inter-node latency coefficient since each recursive-doubling step involves a single exchange rather than a double-binary-tree traversal.
 
**5. Results**
In standalone microbenchmarks on Perlmutter, NVRAR outperforms NCCL by 1.06–1.92x for 256 KB–2 MB messages. For smaller messages (64–128 KB), NVRAR is slightly slower due to three-phase kernel launch overhead, though this overhead does not appear in end-to-end inference workloads. On Vista (single-GPU nodes, InfiniBand), speedups are higher — up to 3.57x for 512 KB messages — because the absence of an intra-node phase eliminates the associated overhead. Integrated into YALIS for decode-heavy inference, NVRAR achieves speedups of 1.14–1.86x for the 70B model and 1.17–1.72x for the 405B model on Perlmutter, and up to 1.92x on Vista. Gains are larger at higher GPU counts and larger batch sizes, where message sizes are more favourable for NVRAR. Performance breakdowns confirm that the speedup comes directly from reduced communication time, with only marginal increases in GPU idle time.
 
**6. Related Work and Conclusion**
Prior work on multi-node inference scaling is limited: vLLM has benchmarked 405B on up to 16 GPUs and Google has studied up to 64-way TP but only on TPUs; no prior work provides systematic performance breakdowns of TP and PP components for large models at scale. Related collective optimisation work includes compression-based approaches (which sacrifice exactness), computation-communication overlap techniques (which address prefill rather than decode), and NVSHMEM-based collectives for mixture-of-experts expert parallelism. NVRAR is unique in targeting exact, latency-optimised all-reduce for the small-message decode regime of tensor-parallel inference. Future work will reduce idle time in NVRAR, address torch.compile OOM failures in YALIS, and auto-tune chunk and block size hyperparameters.
