Prefix and Output Length-Aware Scheduling for Efficient Online LLM Inference
Authors: Inaki Arango, Ayush Noori, Yepeng Huang, Rana Shahout, Minlan Yu
 
**Abstract**
Production LLM inference systems use data parallelism to distribute requests across multiple GPUs, but naive even splitting ignores the fact that many real-world prompts share common prefixes — a property that can be exploited to reuse cached KV state and reduce compute. PREBLE is a prior distributed scheduler that routes prefix-sharing requests to the same GPU, but it approximates per-GPU load using average output lengths, which breaks down when response lengths are highly variable. This paper extends PREBLE with output length-aware scheduling, integrating a per-request output length estimate into the load calculation. Using an enhanced simulation platform, the authors show that this modification reduces average normalised latency by 14.31% at 64 requests per second and 28.89% at 128 requests per second on 8 GPUs. A lightweight BERT-based predictor is also shown to estimate output lengths at 4.8x better-than-random accuracy as a practical replacement for the oracle.
 
**1. Introduction**
LLM inference relies on data parallelism across GPU replicas, with requests distributed to balance load. Two features of modern LLM workloads create opportunities beyond naive round-robin or random dispatch. First, prompts are often much longer than outputs — contexts can reach millions of tokens — making prefill efficiency a key bottleneck. Second, in applications such as chatbots, code assistants, document QA, and agent systems, prompts frequently share prefixes; in some scenarios 85–97% of tokens in a prompt are shared with other prompts. PREBLE exploits prefix sharing by maintaining a global prefix tree and routing requests with matching prefixes to the GPU that already holds the corresponding KV cache. However, PREBLE uses the historical average output length as a uniform proxy for future compute load, which introduces imbalance when output lengths vary substantially across requests. This paper hypothesises and demonstrates that incorporating per-request output length estimates into the scheduling decision improves performance.
 
**2. Related Work**
Prior work on prefix sharing includes hardware-proximal approaches such as HYDRAGEN (lossless shared-prefix attention) and PROMPTCACHE (lossy KV reuse), and inter-GPU schedulers like PREBLE, MEMSERVE, and MOONCAKE. PREBLE remains state-of-the-art for multi-GPU prefix-aware scheduling. Separately, S3 demonstrated that fine-tuning a DistilBERT model to predict decode output lengths enables 6.49x throughput improvement over worst-case benchmarks by enabling shortest-job-first scheduling. TRAIL showed that output length predictions can also be extracted cheaply from the LLM's own layer embeddings. This paper combines the prefix-aware routing of PREBLE with the output length awareness of S3.
 
**3. Results**
 
Characterising PREBLE scalability. The authors extend PREBLE's simulation environment to support overhead tracking, variable output lengths, and experiment caching, then benchmark PREBLE against naive schedulers (random, round-robin, least outstanding requests) across 4, 8, and 16 GPUs at request rates from 1 to 128 requests per second. Two failure modes emerge. First, at high request rates and large GPU counts, PREBLE's scheduling overhead — incurred by its prefix tree matching — grows much faster than naive baselines. On 8 GPUs, PREBLE's overhead rises to 20×10^-4 at 128 RPS versus 3×10^-4 for the least-outstanding-requests (LOR) baseline, and on 16 GPUs it reaches 25×10^-4 compared to 5×10^-4 for LOR. This overhead causes PREBLE to fall behind naive schedulers at 32 RPS on 8 GPUs and at 64 RPS on 16 GPUs. Second, high variance in decode output lengths degrades PREBLE's performance, since its load estimate becomes inaccurate when some requests are far longer or shorter than the historical average.
 
Output length-aware scheduling. To address the output length imbalance, the authors modify PREBLE's E2 scheduler by replacing the average maximum token count stored at each prefix tree node with the true output length of each individual request. This allows per-GPU load calculations to account for actual rather than average decode cost. Using a variance-maximising benchmark — requests are equally likely to have output lengths of 1 or 300 tokens — and an oracle that provides true output lengths, the modified PREBLE achieves 14.31% reduced latency at 64 RPS and 28.89% at 128 RPS on 8 GPUs, with no regression at lower request rates.
 
Output length prediction. As a practical substitute for the oracle, the authors fine-tune the lightweight all-MiniLM-L6-v2 BERT model (6 layers, 80M parameters) on paired (request, output length) data from the Alpaca-52K dataset, framing prediction as a 20-bin classification problem. On a held-out test set the model achieves 4.8x better-than-random performance, with accuracy 0.24, F1 0.22, and AUROC 0.85 — sufficient to guide scheduling decisions in place of exact output lengths.
 
**4. Discussion**
The results confirm that integrating output length awareness into prefix-aware scheduling improves performance in high-demand settings where output length variance is significant. Two directions are identified for future work. First, replacing the oracle with the trained BERT predictor in end-to-end experiments to quantify the performance gap relative to perfect prediction. Second, the authors propose a new scheduling architecture inspired by SPLITWISE that separates prefill and decode phases across different GPU workers, scheduling each in a prefix and output length-aware manner. Single-token activations could predict output lengths for load-aware allocation between prefill and decode pools, potentially reducing the scheduling overhead that limits PREBLE's scalability at high GPU counts and request rates.
