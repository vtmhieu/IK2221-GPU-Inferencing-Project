NEO: Saving GPU Memory Crisis with CPU Offloading for Online LLM Inference
Authors: Xuanlin Jiang, Yang Zhou, Shiyi Cao, Ion Stoica, Minlan Yu
 
**Abstract**
NEO is an online LLM inference system that offloads part of attention computation and KV cache storage from the GPU to the local host CPU, increasing effective batch size and thus inference throughput. Modern inference engines are bottlenecked by GPU memory, leaving substantial compute capacity idle. NEO addresses this with two core techniques: asymmetric GPU-CPU pipelining and load-aware scheduling. Evaluated across T4, A10G, and H100 GPUs with 7B, 8B, and 70B models, NEO achieves up to 7.5x higher throughput than GPU-only approaches while maintaining the same latency, and up to 79.3% gains with more powerful CPUs.
 
**1. Introduction**
LLM inference demands both low latency for user-facing applications and high throughput to make expensive GPU hardware cost-efficient. The KV cache, which grows linearly with sequence length, causes GPU memory to become the binding constraint on batch size, leaving GPU compute underutilised. Prior offloading approaches such as FlexGen and FastDecode either sacrifice latency for throughput or require costly remote CPU servers. NEO instead targets only the local host CPU — resources that come bundled with the GPU at no additional cost — and achieves throughput gains without compromising accuracy or latency.
 
**2. Background and Motivation**
LLM inference consists of a prefilling stage (processing all input tokens to produce the initial KV cache) and a decoding stage (auto-regressively generating output tokens one at a time). Throughput scales almost linearly with batch size up to hundreds of requests, but GPU memory runs out well before that point. The decoding-phase attention operation is memory-bandwidth-bound rather than compute-bound, meaning the performance gap between CPUs and GPUs for this specific operation is much smaller than their raw FLOP counts suggest. For example, an A10G GPU offers 600 GB/s memory bandwidth versus around 200 GB/s for a modern x86 server — a 3x gap, far narrower than the 100x compute gap. This makes CPU offloading of decoding attention viable without causing PCIe bandwidth to become a bottleneck, since communication can be hidden behind GPU computation.
 
**3. NEO Design**
NEO's architecture comprises a CPU-side request scheduler and a set of GPU/CPU executors. The scheduler maintains three queues: a prefilling waitqueue, a GPU decoding runqueue, and a CPU decoding runqueue. Two key techniques drive NEO's design.
 
Asymmetric pipelining addresses the challenge of overlapping GPU and CPU work within each inference iteration. A naive "simple offloading" approach leaves the CPU idle during GPU linear operations, while "symmetric pipelining" (used by prior work like FastDecode) splits the batch evenly but creates three problems: GPU memory underutilisation, load imbalance due to the CPU attention stage taking longer than the GPU linear stage, and difficulty splitting batches evenly with variable request lengths. NEO instead forms two asymmetric sub-batches: batch-0 contains all prefilling requests and GPU decoding requests (long linear stage, short CPU attention), while batch-1 contains the majority of CPU-offloaded decoding requests (short linear stage, long CPU attention). These complementary profiles enable effective overlapping and avoid pipeline bubbles. Layer-wise KV cache swapping further hides PCIe transfer latency by initiating transmission immediately after each layer's KV values are computed.
 
Load-aware scheduling handles dynamically changing real-world workloads. At each iteration, NEO follows four principles: greedy selection between GPU-only and two-batch pipelining schedules based on estimated throughput; balancing GPU and CPU busy times to minimise pipeline idle periods; ensuring the CPU is never busy while the GPU is idle; and maximising the number of requests packed into each batch. The scheduling overhead is under 3ms per iteration, negligible relative to typical iteration times of hundreds of milliseconds.
 
**4. Implementation**
NEO is implemented on top of SwiftLLM, a streamlined vLLM-equivalent system, in approximately 4,000 lines of Python and 1,500 lines of C++. A custom paged attention CPU kernel library (PACPU) is implemented using the Intel ISPC compiler, enabling efficient SIMD memory access and portability across AVX2, AVX512, and ARM Neon instruction sets. To reduce Python GIL-related kernel launching overhead, most Triton-JIT GPU kernels were replaced with custom CUDA C++ kernels, cutting per-layer launching overhead from 1.2ms to 0.6ms on Llama-3-8B. Multi-GPU tensor parallelism is supported via Ray actors and PyTorch's NCCL wrapper.
 
**5. Evaluation**
NEO is evaluated on AWS EC2 instances (T4, A10G) and a local 8×H100 server using the Azure LLM coding trace and OpenAI summarisation dataset, with Llama-2-7B, Llama-3.1-8B, and Llama-3.1-70B models. NEO sustains higher request rates than vLLM at the same latency across all settings: 14.3% higher throughput on H100, 6.4% on A10G, and 563% on T4, where the GPU's severely constrained KV cache memory makes offloading especially impactful. Latency distributions are comparable to vLLM at all percentiles. Compared to FastDecode+, NEO achieves lower latency and higher throughput because its scheduler can fall back to GPU-only mode when offloading is not beneficial, whereas FastDecode+ is forced to use the CPU even when requests are short and the CPU becomes the bottleneck. Sensitivity analysis shows throughput gains of 12–79% on A10G depending on CPU memory bandwidth, confirming that memory bandwidth rather than core count is the determining factor for CPU attention performance.
 
**6. Discussion**
NEO is particularly well-suited to scenarios where GPU compute capacity is growing faster than GPU memory — for instance, H100 triples the compute of A100 while retaining the same 80 GB memory. Compared to chunked-prefill techniques like Sarathi-Serve, NEO avoids the repetitive KV cache reloading that chunked-prefill incurs and performs better on memory-constrained GPUs. The two approaches are complementary and could be combined. Future directions include offloading dense operations to the CPU for workloads with very short outputs, extending to remote CPU workers for additional throughput, and formally studying energy consumption trade-offs. NEO degrades gracefully to GPU-only mode when the GPU has sufficient memory to saturate its compute without offloading.
 
**7. Related Work**
NEO builds on GPU-efficient inference systems such as vLLM, Orca, SGLang, and Sarathi-Serve, incorporating techniques like selective batching and paged attention. It differs from offloading-focused systems (FlexGen, HeteGen, PowerInfer, TwinPilots) in targeting online latency-sensitive inference rather than offline throughput maximisation, and from FastDecode in using partial rather than full offloading to avoid CPU bottlenecks. Unlike quantisation and sparsification approaches (AWQ, SparseGPT, GPTQ), NEO achieves performance gains without any reduction in model accuracy.
 
**8. Conclusion**
NEO demonstrates that the local host CPU, typically underutilised during GPU inference, can be leveraged to meaningfully increase batch sizes and inference throughput without sacrificing latency or accuracy. Through asymmetric pipelining and load-aware scheduling, NEO achieves up to 7.5x throughput gains over GPU-only systems at the same hardware cost, with code open-sourced to support further research into cost-efficient LLM inference.
